# 权限策略

## 1. 权限边界定义

### 1.1 操作分类

#### 🟢 可以自主执行的操作（无需确认）

**文件操作**
- 读取文件内容
- 列出目录结构
- 搜索代码
- 检查诊断信息

**代码检查**
- 运行类型检查（npm run type-check）
- 运行代码检查（npm run lint）
- 运行Django检查（python manage.py check）
- 调用TRAE-code-review进行代码审查

**开发工具**
- 调用相关开发Skill（backend-developer、frontend-development等）
- 生成测试代码
- 运行安全的构建命令

**信息获取**
- GetDiagnostics
- SearchCodebase
- Grep
- Glob
- LS

#### 🟡 需要人工批准的操作（必须确认）

**创建文件**
- 创建新的源代码文件
- 创建配置文件
- 创建文档文件
- 创建项目结构

**修改重要文件**
- 修改package.json
- 修改settings.py
- 修改tsconfig.json
- 修改Prisma schema
- 修改核心配置文件

**删除操作**
- 删除任何文件（除非用户明确说"删除xxx"）
- 删除目录

**运行命令**
- 可能修改系统的命令
- 数据库修改命令
- 安装新依赖

**破坏性变更**
- 修改API接口
- 修改数据库结构
- 重构核心逻辑

#### 🔴 必须停止并报告的操作（禁止执行）

**Git操作**
- git commit
- git push
- git pull
- git checkout
- git merge
- git rebase
- 所有其他git命令

**敏感操作**
- 修改.gitignore
- 修改.git目录
- 读取或提交.env文件内容
- 记录或泄露敏感信息（密码、API密钥等）

**高危命令**
- 删除系统文件
- 修改系统配置
- 强制覆盖操作
- 未知或危险的命令

## 2. 决策流程

### 2.1 操作决策树

```
开始
  │
  ▼
这是什么类型的操作？
  │
  ├─ 🔴 禁止操作 → 停止 → 报告用户 → 等待指示
  │
  ├─ 🟡 需要批准 → 询问用户 → 用户同意 → 执行
  │                               │
  │                               └─ 用户拒绝 → 停止
  │
  └─ 🟢 自主执行 → 检查安全 → 执行 → 验证结果
```

### 2.2 询问用户的格式

当需要用户批准时，使用以下格式：

```
⚠️ 需要您的确认

**操作描述**：[简要描述要执行的操作]

**涉及文件**：
- [文件1]
- [文件2]

**风险评估**：[低/中/高]

**选项**：
A. 确认执行
B. 取消操作
C. 修改方案（请说明）

请选择：
```

## 3. 报告格式

### 3.1 错误报告

当遇到禁止操作或错误时：

```
❌ 操作被阻止

**操作类型**：[被阻止的操作]
**原因**：[为什么被阻止]
**规范依据**：[引用相关规范文件]

**建议**：
- [建议1]
- [建议2]

请您手动执行此操作，或告诉我其他方案。
```

### 3.2 成功报告

```
✅ 操作完成

**执行内容**：[简要描述]
**涉及文件**：[文件列表]
**检查结果**：[通过/有警告]
**下一步**：[建议]
```

## 4. 权限升级路径

### 4.1 升级触发条件

当遇到以下情况时，需要升级权限：
- 用户明确要求执行原本禁止的操作
- 紧急情况需要特殊处理
- 重复请求同一操作

### 4.2 升级流程

```
1. 记录用户请求
2. 评估风险
3. 确认用户完全理解后果
4. 仅执行用户明确要求的部分
5. 详细记录所有操作
6. 完成后立即恢复权限限制
```

### 4.3 例外记录

所有权限例外必须记录在案：

```
📝 权限例外记录

**日期**：YYYY-MM-DD
**用户**：[用户]
**操作**：[执行的操作]
**原因**：[为什么需要例外]
**结果**：[结果如何]
```

## 5. 违规处理

### 5.1 违规类型

1. **轻度违规**：执行了需要批准但未确认的低风险操作
2. **中度违规**：执行了有风险的操作
3. **重度违规**：执行了明确禁止的操作（如git操作）

### 5.2 处理流程

1. **立即停止**：停止所有相关操作
2. **报告用户**：详细说明发生了什么
3. **评估影响**：评估可能造成的影响
4. **修复措施**：提供修复方案
5. **防止再发**：更新规范或检查清单

### 5.3 降级措施

- 限制操作范围
- 增加确认要求
- 暂停某些功能
